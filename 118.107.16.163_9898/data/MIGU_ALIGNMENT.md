# 业务逻辑向咪咕靠齐 - 对齐清单

目标：所有业务逻辑尽量向咪咕（参考：咪咕后台 imxehc.com/new/admin-panel 等）靠齐，包括菜单结构、文案、订单流程、资金与抽佣、通知与权限等。

---

## 〇、咪咕业务模式（当前实现）

**是的：咪咕侧是「集成多条游戏充值通道，再通过统一 API 分发给下游」；下游可以是商户或四方，对接的每条通道内部再对接到具体四方/直连渠道。**

- **聚合层（9898 + xnapi）**  
  - 在后台配置多条**游戏渠道**（Access），每条渠道下有多个**游戏账号**（Account）作为出款池。  
  - 每条渠道对应一个 **module**（如 Bilibili、Wanmei、Jvren、Huya 等），即 xnapi 里对**某一套四方或直连渠道**的对接实现。  
  - 下游（商户或四方）只调用咪咕的统一 API（如 `/api/pay/bulid`、`/ev/{sha1}` 等），传 appid、金额、支付方式等。

- **路由与执行**  
  - 收到下单请求后，按 user、金额、pay_type、可选 code 等条件，在 **Account::queryMay** 里筛出可用通道（access）和账号（account）。  
  - **Order::match** 按顺序用每个账号所属 access 的 **module** 调对应 `build()`，向该通道背后的四方/直连发起下单；成功则落单并返回支付链接。  
  - 回调、查询、结算等同样按订单所属的 access/account 找到对应 module 处理。

- **结论**  
  - 咪咕 = **内部集成多套游戏充值通道**（每套可对接一个四方或直连），**通过 API 对下游提供统一入口**；下游不直接选通道，由咪咕内部选路到具体通道（access + account）再对接四方。  
  - 若下游本身是「四方」，则四方调用咪咕 API 拿单/拿链接，咪咕再把自己的多条通道（每条可能对接着不同上游四方）分配给这笔请求。

## 二、业务逻辑对齐要点（与咪咕一致）

- **订单生命周期**：创建 → 待支付 → 已支付/已回调 → 结算/分账 → 完成/关闭；状态与操作与咪咕后台语义一致（展示与筛选保持「待支付」「已支付」等）。
- **资金与抽佣**：资金流水与系统扣佣记录分离展示；抽佣记录只读、不在此处做人工调账；结算口径与咪咕一致（按订单完成、按渠道/账号汇总等）。
- **收银台**：以「收银台地址」为统一入口，不再保留与收银台重复的旧支付页；新支付链接以收银台为准。
- **通知**：新订单以顶栏「新订单通知」+ 红点为主，逻辑与咪咕一致（近期待支付数量、点击进订单）。
- **菜单与文案**：后台菜单、列表表头、按钮、提示语等尽量使用咪咕体系用语（游戏渠道、游戏账号、支付订单数据、收银台地址、抽佣点位记录、分后台管理等），见 application/admin/lang/zh-cn/migu.php 与 data/migu_menu.sql。

## 四、执行与维护

- **首次/新环境**：执行 `data/migu_menu.sql`（注意表前缀），清空缓存后刷新后台，即可得到咪咕化菜单与站点名。
- **新增菜单**：新增 yunos 相关菜单规则时，在 migu_menu.sql 中增加对应 `UPDATE fa_admin_rule SET title = '咪咕文案' WHERE name = 'yunos/xxx'`，并在 `application/admin/lang/zh-cn/migu.php` 中增加键值，便于全局替换与多语言扩展。
- **业务规则变更**：订单状态、资金结算、抽佣规则等若有调整，优先对照咪咕后台行为，保持「业务逻辑尽量像咪咕靠齐」。

以上为当前「业务逻辑向咪咕靠齐」的完整清单与维护说明。
